## Context

The Bazodiac frontend is a Vanilla ESM SPA where each page module under `public/src/pages/` must follow a handful of contracts so that animations, navigation cleanup, and visual consistency keep working:

- **`bz-h1` design-token class** on the hero `<h1>` (Cormorant Garamond serif via `tokens.css`/`main.css`)
- **`data-section` attributes** on major content sections (hero, content, etc.) — Playwright contract anchor
- **`data-page-title` marker** on the hero element so `wireHeroRolling` finds it
- **`wireHeroRolling(app)` invocation** + return of its cleanup callback so the hash-router can deterministically stop the RAF loop
- **No fake/hardcoded astrological data** — pages render API output, not baked-in strings

This skill mechanizes the audit that I6/I7 ran manually: walk the list of pages, build the contract-compliance table, identify gaps, propose targeted fixes per gap. It does NOT implement the fixes — that is the job of `/writing-plans` + `/executing-plans` (or `/review-fix-cycle`).

See `docs/components-contract.md` for the canonical contract definitions.

## Your Task

Run the page-contract audit and produce an `N × M` evidence table where N = pages and M = contract dimensions.

### Steps

1. **Determine audit scope** via AskUserQuestion:
   - Default: all pages under `public/src/pages/`
   - Alternative: a comma-separated subset (e.g. `BaziPage.js, WesternPage.js`)
   - Whether to also include hidden routes (`/love`, `/personality`, `/career-finance`, `/dashboard`, `/transit-calendar`)

2. **Dispatch a single read-only Agent** (`caveman:cavecrew-investigator`) to read each page and report per-page facts:

   ```
   For each page in <scope>, report:

   1. Does it use the `bz-h1` class on a heading? (yes/no + line)
   2. Does it set any `data-section` attribute? (yes/no + values found)
   3. Does it emit a `data-page-title` marker in its innerHTML template? (yes/no + line)
   4. Does it import `wireHeroRolling` from '../components/RollingText.js'? (yes/no)
   5. Does it call `wireHeroRolling(app)` (or equivalent)? (yes/no + line)
   6. Does the page function `return` the cleanup callback at its terminal statement? (yes/no + line)
   7. Does it handle empty/null profile state (UnavailableCard or early-return)? (yes/no + how)
   8. Approximate line count.

   Output one compact table per file, then a cross-file summary of the
   top 3 contract gaps across all pages.
   ```

3. **Render the audit document** at `docs/qa/<YYYY-MM-DD>-page-contract-audit.md` with:
   - The per-page evidence table (from the agent)
   - A **gap summary**: which contract each page violates, ranked by severity (missing `data-page-title` is high — breaks hero animation; missing `data-section` is medium — breaks Playwright contracts)
   - A **proposed remediation matrix**: page × gap × suggested fix (e.g., "WuxingPage: add `data-page-title` to hero `<h1>` and capture wireHeroRolling cleanup at end of function")

4. **Stop and present.** Do NOT modify any page files. The output is an audit; remediation goes through `/writing-plans` next.

### Guardrails

- **Read-only.** Never edit `public/src/pages/*.js` from this skill. Even small-looking fixes go through a separate plan-and-execute cycle.
- **No hardcoded contract.** Read the actual contract from `docs/components-contract.md` if it exists; only fall back to the defaults in step 2 if the doc is missing.
- **Surface inconsistencies, not gut feelings.** Each finding must reference a line number or an explicit absence (`no <pattern> in file`).
- **Don't conflate `data-lane` with `data-section`.** They serve different purposes (`data-lane` = color theming; `data-section` = content hierarchy anchor).
- **Skip MethodPage's async cleanup** — it's a known exception (async function returns Promise; relies on `isConnected` self-cancel). Note it as accepted, not as a gap.

### Variants

- `--quick` — skip the agent dispatch; do source-text grep only via Bash for the 5 binary checks. Faster, less precise on edge cases.
- `--with-fixtures` — also report whether each page has a corresponding test file under `test/<page-slug>.test.js`.
