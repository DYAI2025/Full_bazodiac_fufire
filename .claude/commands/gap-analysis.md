## Context
Project: Bazodiac / FuFirE proxy. Backend in `server.js`, frontend in `public/src/pages/`. The API delivers western + BaZi + fusion + synastry + daily data. Pages sometimes show "no data" for fields the API actually provides.

## Your Task

Produce a structured gap analysis: what does the API deliver vs what does each UI page actually render?

### Step 1: Map available data

```bash
# What endpoints exist
grep -n "orchestrate\|'/api/" server.js | head -40

# What fields normalizeAzodiacResult produces
grep -n "vm\.\|\.western\.\|\.bazi\.\|\.fusion\." server.js | grep "=" | head -50
```

### Step 2: Map what each page renders

For each page in `public/src/pages/`:
- What ViewModel fields does it read?
- What fields does it reference that might be undefined/null?
- Are there `|| 'keine Daten'` or `?? '—'` fallbacks hiding gaps?

```bash
grep -rn "keine Daten\|no data\|kein\|undefined\|?? '—'\||| ''" public/src/pages/ | head -40
```

### Step 3: Map domain projections

```bash
grep -n "export function\|return {" public/src/domain/projections.js | head -30
```

### Step 4: Produce gap table

For each page:
```
| Page | Expected fields | Actually rendered | Gap |
```

Rate each gap:
- **CRITICAL** — field exists in API, displayed as error/empty
- **MISSING** — field exists but page doesn't render it at all
- **OK** — working correctly
- **UNKNOWN** — needs live API test to confirm

### Step 5: Recommend actions

Group by effort:
- Quick wins (1 line fix, projection or template change)
- Medium effort (new component or domain transform)
- Long effort (new endpoint or normalizer work)

### Guardrails
- Never assume a gap is fixed until confirmed with live `/api/azodiac/profile` call or test
- Mark assumptions as ASSUMPTION, not confirmed facts
